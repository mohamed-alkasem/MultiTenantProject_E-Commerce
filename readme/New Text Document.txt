# ملخص مشروع MultiTenantStore

المشروع عبارة عن منصة **Multi-Tenant E-Commerce SaaS**.
الفكرة أن صاحب المنصة يسمح للتجار بإنشاء متاجر إلكترونية خاصة بهم، وكل متجر يكون له بياناته المستقلة وقاعدة بيانات منفصلة.

المعمارية المعتمدة هي:

```text
Onion Architecture / Clean Architecture Style
```

أي أن النظام مقسّم إلى طبقات واضحة:

```text
Domain
Application
Persistence
Infrastructure
Web
Tests
```

والاعتماد يكون دائمًا من الخارج إلى الداخل:

```text
Web → Application
Infrastructure → Application
Persistence → Application / Domain
Application → Domain
Domain → لا يعتمد على أحد
```

# 2. طريقة تحديد الـ Tenant

اتفقنا أن تحديد المتجر الحالي يتم بطريقتين فقط حاليًا:

## Public Store

للمتجر العام، يتم تحديد المتجر من الـ subdomain:

```text
ahmad.platform.com
```

النظام يقرأ:

```text
subdomain = ahmad
```

ثم يبحث في جدول:

```text
StoreDomain
```

وبعدها يجيب:

```text
StoreId
StoreDatabase
ConnectionStringEncrypted
```

ثم يفتح قاعدة بيانات هذا المتجر.

## Dashboard

بالنسبة لصاحب المتجر أو الموظفين داخل لوحة التحكم، يتم تحديد المتجر من الـ claims داخل الـ JWT token:

```text
store_id
store_role
user_id
```

ثم النظام يتحقق من جدول:

```text
StoreUser
```

ليتأكد أن المستخدم له صلاحية على هذا المتجر.

قررنا حاليًا عدم استخدام:

```text
Custom Domain
Header Tenant
Route Tenant
```

وقد تؤجل هذه الميزات لاحقًا.

# 3. الـ Middleware المتفق عليه

أهم middleware في النظام هو:

```text
TenantResolutionMiddleware
```

وظيفته:

```text
1. يقرأ نوع الـ route.
2. إذا كان route لا يحتاج tenant يتجاوزه.
3. إذا كان Dashboard يقرأ tenant من Claims.
4. إذا كان Public / Customer / Cart / Checkout يقرأ tenant من Subdomain.
5. يجلب TenantInfo من Main DB.
6. يتحقق أن المتجر Active.
7. يخزن معلومات المتجر في CurrentTenant.
8. بعدها TenantDbContext يستخدم connection string الصحيح.
```

ترتيب الـ pipeline المتفق عليه:

```text
ExceptionHandling
StaticFiles
Routing
Authentication
TenantResolution
Authorization
Controllers / MVC Routes
```

والسبب أن `TenantResolution` يأتي بعد `Authentication` هو أن dashboard يحتاج قراءة الـ claims من المستخدم.

# 7. ملاحظات مهمة على تصميم قاعدة البيانات

اتفقنا على هذه النقاط:

```text
1. لا نجعل Entity classes sealed.
2. نستخدم AuditableEntity و ISoftDelete.
3. أغلب الحقول مثل UpdatedAt و DeletedAt تكون nullable.
4. ProductVariantId في CartItem و OrderItem يكون nullable.
5. Order يحتوي address snapshot.
6. OrderItem يحتوي product snapshot.
7. Invoice و InvoiceItem يحتويان snapshots كذلك.
8. Payment علاقة one-to-many مع Order.
9. StoreDatabase يحتوي ConnectionStringEncrypted ولا يظهر في أي API response.
```

# 14. Migrations

عندنا نوعين migrations:

## MainDbContext Migration

خاص بالـ platform database.

الأمر:

```powershell
dotnet ef migrations add InitialMainDb `
  --project .\src\MultiTenantStore.Persistence `
  --startup-project .\src\MultiTenantStore.Web `
  --context MainDbContext `
  --output-dir Migrations\Main
```

ثم:

```powershell
dotnet ef database update `
  --project .\src\MultiTenantStore.Persistence `
  --startup-project .\src\MultiTenantStore.Web `
  --context MainDbContext
```

## TenantDbContext Migration

خاص بجداول كل متجر.

الأمر:

```powershell
dotnet ef migrations add InitialTenantDb `
  --project .\src\MultiTenantStore.Persistence `
  --startup-project .\src\MultiTenantStore.Web `
  --context TenantDbContext `
  --output-dir Migrations\Tenant
```

لأن TenantDbContext يعتمد على connection string ديناميكي، أضفنا design-time factory:

```text
TenantDbContextFactory
```

تقرأ connection string:

```text
TenantMigrationDb
```

هذه القاعدة فقط من أجل design-time migration، وليست قاعدة متجر حقيقية.

لاحقًا عند إنشاء متجر جديد، النظام يستخدم:

```text
TenantDbTemplate
```

ويستبدل:

```text
{databaseName}
```

باسم قاعدة بيانات المتجر ثم يشغل:

```text
Database.Migrate()
```

# 15. مشاكل تم حلها أثناء الإعداد

واجهنا مشاكل وتم حلها:

## اختلاف TargetFramework

كانت بعض المشاريع `net8.0` وبعضها `net9.0`.
اتفقنا على توحيدها كلها، وعمليًا كان المشروع يستخدم packages 9.0.15.

## IdentityUser غير معروف داخل Domain

لأن `ApplicationUser` يرث من:

```csharp
IdentityUser<Guid>
```

احتاج مشروع Domain إلى package:

```text
Microsoft.Extensions.Identity.Stores
```

## Microsoft.Extensions.Options.ConfigurationExtensions

احتجنا إضافة package مطابق للنسخة:

```text
Microsoft.Extensions.Options.ConfigurationExtensions 9.0.15
```

لتفعيل:

```csharp
services.Configure<TOptions>(configuration.GetSection("..."))
```

## Microsoft.CodeAnalysis.Workspaces

وقت migration ظهر نقص في Roslyn package، وتم حله بإضافة:

```text
Microsoft.CodeAnalysis.CSharp.Workspaces 4.8.0
```

إلى Persistence و/أو Web.

## TenantDbContext design-time error

لأن EF لم يكن يعرف كيف ينشئ TenantDbContext، أضفنا:

```text
TenantDbContextFactory : IDesignTimeDbContextFactory<TenantDbContext>
```

## SetBasePath / AddEnvironmentVariables errors

تم حلها بإضافة packages المطلوبة أو حذف `AddEnvironmentVariables()` مؤقتًا.
